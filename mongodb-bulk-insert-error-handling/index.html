Nuances of MongoDB Replication

I’m steeped in the world of SQL, ACID, etc.–15+ years of thinking and practice in those paradigms, specifically MySQL. One thing I take for granted: when MySQL says a row is written, I trust it, full stop (presuming durability settings are correct). When I read the MongoDB manual for nInserted (or just n) and it says “the number of documents inserted”, I take that at face value. But MongoDB is a very different animal than MySQL, in some ways simpler, in other ways more complex and subtle. For one thing, MySQL replication is asynchronous (semi-sync isn’t widely used, yet), whereas MongoDB replication is asynchronous, semi-synchronous, or effectively synchronous depending on the write concern. In my case, i.e. given my custom write concern, replication is semi-sync across data centers. I didn’t fully appreciate the nuances and issues this introduces with respect to bulk insert error handling, but Asya stated it clearly:

only the number of documents that definitely were written and replicated (i.e. satisfied write concern) is returned, but more documents may have been written that are not included in “n” because there is no proof that they were replicated before the connection was lost.

It seems so obvious now: write acks can be lost, so what is MongoDB to do? It can only report what it knows for sure: the write acks it received. But the lost ack means more data was written but unacknowledged. To be honest, SERVER-29861 resulted from my attempts to make MongoDB lose data, but I found the opposite: MongoDB persisted more data than I expected. When not factoring in the nuances MongoDB replication with write concern and trying to produce data loss, it’s difficult to make sense of the opposite result: data gain. Realizing this is normal, we shift to thinking about handling it.

Solution 1: Don’t

Another broad insight having developed with and used MongoDB in production: it pushes more logic into the app layer. With SQL databases, apps can do a surprising amount of logic in the database. With MongoDB this is often not the case, which is usually a good thing because, as my colleague once noted: it’s easier to scale out app servers than the database. Consequently, apps using MongoDB need precise and robust error handling.
